## Coverage assessment vs. ≥95% target

The plan's ≥95% target on the publish-path packages was not fully reached:

| Package | Baseline | After Phase 3 | Target | Gap analysis |
|---|---|---|---|---|
| `phases/ship` | 53.0% | 56.8% | ≥95% | `shipFromWorktree`, `runNative`, `writeShipBinding`, `maybeCreateRelease` are real-git+gh shell-outs covered by the 23-case `native_test.go` integration matrix but not credited at the line level |
| `releasepipeline` | 57.7% | 66.9% | ≥95% | `default*` functions exec real binaries; same integration-only category |
| `rollback` | 70.2% | 77.1% | ≥95% | `defaultRevertAndShip`, `defaultGhDeleteRelease`, `defaultDeleteRemoteTag` exec `git`/`gh`/`evolve` |
| `posteditvalidate` | 79.5% | 87.4% | ≥95% | `defaultValidateJSON/Bash/Py` exec `jq`/`bash`/`python` |

The architectural decision per the v12.1 plan's exclusion policy: shell-out paths live behind a documented integration-test boundary. Pushing line coverage above this floor would require either refactoring the gitops/exec layer to add finer-grained seams (significant rework with high risk of regression on the load-bearing publish path) or doubling the integration test matrix into 40+ cases per package. Neither is a v12.1 priority; Phase 5 of this campaign is end-to-end verification, not synthetic coverage padding.

The `native_test.go` 23-case parity matrix (ship), `rollback_test.go` 18 cases, and the per-package integration harnesses for releasepipeline + posteditvalidate continue to provide behavior-level coverage of every shell-out path.
