# ADR-0060: Fully data-driven transition kernel (validator-anchored)

Status: Accepted
Date: 2026-06-22
Supersedes: ADR-0058 decisions §1 (legality graph hardcoded) and §8 (artifact map hardcoded). Extends ADR-0058's slice campaign (S1–S6 shipped).
Campaign: PA-DDK (plan: `buzzing-shimmying-quasar`).

## Context

ADR-0058 made the orchestrator transition kernel *partly* config-driven (audit verdict branch, retro/debugger gates, mandatory-anchor set/order) while deliberately keeping the **legality graph** (`allowed`) and the **artifact-gate map** (`anchorArtifactPresent`) hardcoded as config-independent trust anchors — so config could only *narrow* legality, never widen it. ADR-0058 explicitly **rejected** "Alternative C: full generic data-driven FSM" because a bare generic FSM lets a malformed/malicious config widen the legality oracle and bypass the non-gameable ship floor.

The operator requirement has since hardened: **every phase name, function, and setting must be extracted to dynamic configuration** — zero hardcoded phase identity may drive behavior in production *or* test code. The kernel must become a generic engine; an operator changes the entire flow by editing the registry, not Go. This is Alternative C.

The blocker to Alternative C was always safety, not feasibility. Source review confirms the floor's *true* root of trust is **runtime, artifact-backed**: `SpineSatisfiedUpTo` reading real on-disk digests, the audit verdict routing in `Next`, and the ship-time tree-SHA binding + EGPS `red_count==0`. The hardcoded legality graph is only a *structural backstop*, not the load-bearing guarantee.

## Decision

**1. Config-drive the entire flow** — legality graph (`legal_successors`), artifact gates (`gate`), recovery targets (`recovery`), early-exit (`early_exit`), verdict branches (`on_pass`/`on_fail`, already live), branching strategy (already live), aliases — all declared in the registry. The Go kernel reads them; it holds no phase-name `switch` in flow logic. Most are *per-phase* descriptor fields; the **legality graph is the one exception** — see the DDK-5 note below.

**1a. Legality graph lives in the `config` block, not a per-phase field (DDK-5 implementation note).** The graph's nodes are not all operator-facing phases: `start`/`end` are graph terminals (no phase entry) and `debugger` lives in the kernel `builtinControlSpec` seam (not the registry `phases[]`). A per-phase `legal_successors` field would orphan exactly the sentinel edges (`start→{intent,scout}`, `ship→end`, `debugger→…`) that anchor I9 reachability. So the graph is a single `config.legal_successors: { name: [successors] }` SSOT keyed by phase NAME (sentinels included) — mirroring the existing `config.spine_order` / `config.mandatory_phases` cross-phase config. Names denormalize through `phaseFromRouter` (so `retrospective`→`retro` etc.); an absent map keeps the kernel's literal `allowed` graph (byte-identical fallback). The validator (`ValidateSafetyInvariants`) is wired as a HARD load-time gate at the composition root — DDK-1 landed it dark (computed, never enforced); DDK-5 makes it fail-closed at the `RunCycle`/`RunCycleFromPhase` chokepoint, because that is the slice where the graph becomes operator-mutable and the validator becomes the floor's sole structural guarantee.

**2. Relocate the trust anchor from the graph to a validator.** A hardcoded, **phase-agnostic** `ValidateSafetyInvariants(graph, cfg, catalog)` runs at load and HARD-FAILS any config whose flow could ship without the floor. Its invariants are quantified over config *roles* (`M = mandatory anchors`, `F = ship floor ∪ evaluator`, `E = evaluate-archetype phases`, `effectiveOrder` positions) — **never phase-name literals** — so renaming `scout`→`recon` or `audit`→`evaluate` does not weaken them. The validator, not the graph literal, is now the thing config cannot move. This is Alternative C **plus** the validator that Alternative C lacked — sound where bare-C was not.

**3. The digest stays trusted Go.** `router.RoutingSignals` (how on-disk handoff artifacts are read into objective signals) is the anti-spec-gaming verification root and remains code. Only the gate *thresholds* (`requires_present`, `verdict_in`) become config; the gate is *evaluated* against the trusted digest.

**4. Floor invariants (the validator's checks).** I1 every start→ship path traverses every `m∈M` in order (each dominates `ship`); I2/I3 some evaluate phase with a verdict-gate `⊆{PASS,WARN}` dominates `ship` and every direct ship-predecessor; I4 build→evaluate→ship reachable order only; I5 `→end` only before the first floor phase; I6 no edge skips an anchor; I7 single ship sink, canonical verdicts, floor-gate `⊆{PASS,WARN}`; I8 every `on_pass`/`on_fail`/recovery target ∈ that phase's `legal_successors`; I9 reachability + **`F ⊆ M`** (without which the runtime artifact gate goes inert).

**5. Acceptance = semantic-equivalence + a never-diff floor property.** Not strict byte-identity (ADR-0058's per-slice oracle relaxes to semantic-equivalence, already accepted in S6). A re-derived oracle (derivation + snapshot-golden + floor-property over a checked-in reference registry) guards each slice; the floor property `ship ⇒ scout ∧ build ∧ shippable-audit` may never diff.

## Why this is safe (bypass analysis summary)

With the validator at load AND the unchanged runtime layers (verdict routing, artifact gate, `CanTerminateEarly`, `ClampPlanToFloorWith`, ship tree-SHA binding): structural attacks (add `scout→ship`, drop audit, accept FAIL, early-exit post-build, shrink the floor) are **rejected at load** by I1–I9; runtime lies (claim PASS with no artifact) are **rejected by the artifact-backed digest + tree-SHA binding**, which never depended on the hardcoded graph. The floor degrades gracefully — no single point's failure ships unaudited code. This is strictly *stronger* than the hardcoded graph, which only guaranteed *specific literals*; the validator guarantees the *property* for any renaming.

## Consequences

Positive: the kernel is a generic engine; changing the flow is a registry edit; the floor guarantee is an explicit, tested, phase-agnostic property rather than an implicit literal.

Negative / accepted: the validator becomes a new CI-critical trust component (mitigated by adversarial broken-registry tests + a guard test on the shipped registry); a new test-fixture indirection (a reference registry + accessor loader) replaces inline phase constants in kernel tests.

## Alternatives considered

- **Keep the legality graph hardcoded as the one safety oracle** (config-drive only settings). Rejected by the operator requirement (the graph is phase identity in code). Safe but does not meet "everything in config."
- **Also config-drive the digest extraction.** Rejected: expands the trust surface of the anti-gaming root; the digest reads objective artifacts and must stay trusted Go.

## Rollout (PA-DDK slices)

DDK-0 land ADR-0058 S6; **DDK-1 (this ADR) + the validator over today's checkable invariants (additive)**; DDK-2 reference fixture + re-derived oracle; DDK-3 spineOrder→config; DDK-4 artifact-gate→config (+gate invariants); DDK-5 legality graph→config (+dominance invariants, validator-gated); DDK-6 recovery maps→config; DDK-7 early-exit + router floor→config; DDK-8 alias/name-skew→config (terminal, grep-clean). Each slice is an independent PR from its own worktree, oracle-green, floor-property-green.

## Campaign closure (2026-06-22)

The config-driven transition **flow** is complete and merged: **DDK-1** (validator, dark), **DDK-2** (reference fixture + re-derived oracle), **DDK-3** (spine order), **DDK-4** (artifact gates), **DDK-5** (legality graph + validator wired as the HARD load-time gate at the `RunCycle`/`RunCycleFromPhase` chokepoint), **DDK-6** (recovery targets), **DDK-7** (early-exit), **DDK-7b** (source-write axis). The kernel is a generic engine: an operator changes phases, order, legality, verdict/branch/gate semantics, recovery targets, early-exit, and the write axis by editing `phase-registry.json` — never Go. Each behavior has a literal fallback so a catalog-less state machine and a registry omitting a field stay byte-identical (the transition oracle proves it), and `ValidateSafetyInvariants` (phase-agnostic, role-quantified) is the relocated trust anchor, fail-closed before any phase runs. **DDK-8** (alias/name-skew) is resolved as a *decided boundary*, not a code change: the grep-clean goal is met (no phase-name literal drives a transition), and the residual core↔registry vocabulary skew is bridged by the centralized converters (see Deliberate boundary). The campaign has no open items.

**Deliberate boundary — what stays Go, and why (the residual literals are NOT flow):**

- **The ship-floor default** (`router.DefaultShipFloor` = `{tdd,build,audit}`). The floor is *already* operator-configurable through `.evolve/policy.json:ship_floor` → `policy.FloorPhases()` → `WithShipFloor` → `o.shipFloor` → `resolvedShipFloor()`. The Go literal is the documented **safe default** taken only when no policy override exists — identical in role to the spine/legality literal fallbacks. Adding a second registry source (`config.ship_floor`) would **duplicate** the policy.json surface (violates the project's no-duplication rule) for no operator gain.
- **`EvaluatorFloorPhase` (`"audit"`)** — the single non-removable evaluator, re-asserted by `ClampPlanToFloorWith` even for a caller that bypasses policy. It is intentionally duplicated in `policy.go` to avoid an import cycle (each layer independently guarantees an evaluator — defense in depth) and pinned by `TestEvaluatorFloorPhase_SingleSource`. Config-driving it would dissolve the self-sealing guarantee; the validator already enforces the evaluator-present property over config, so the const is a backstop, not the gate.
- **The `retro`↔`retrospective` / `tdd-engineer`→`tdd` alias bridge** (`phaseFromRouter` / `canonicalCatalogName` / `normalize`) — an **internal** translation between the Go enum (`core.PhaseRetro` stringifies `"retro"`) and the registry vocabulary (`"retrospective"`), plus a role→phase fold. An operator cannot reconfigure `core.PhaseRetro`, so this is not an operator setting; the three functions are the centralized converters. (The deeper root-cause fix — renaming `core.PhaseRetro`'s wire string to `"retrospective"` — is now **decided against**, not merely deferred. `"retro"` is the *serialized* phase identity in `state.json` and `ledger.jsonl`, pinned by the byte-identity guard `cyclestate.TestPhaseConstants` precisely because such a drift is a trust-kernel break; the rename would therefore need a permanent legacy-compat shim at every load boundary plus a ~68-test-file churn, and would buy only grep-cleanliness — which the *flow* already has (no phase-name literal drives a transition after DDK-1…7b). Per the project minimalism rule, a working centralized converter is not replaced for cosmetics. The bridge is the accepted permanent boundary.)
- **`isFloorPhase`** (`{intent,scout,tdd,build,audit,ship}`) — the insertion-cap exemption set in the router walk, distinct from both `cfg.Mandatory` and `cfg.SpineOrder`; an optimization, not a flow or safety gate.
- **Kernel test vocabulary.** Tests that read the registry assert through the `kerneltest` fixture's structural accessors (zero hardcoded names — a registry rename needs no test edits). Bare-state-machine and synthetic-config algorithm tests deliberately use the Go `Phase` constants with controlled inputs: they exercise Go literals/algorithms, not the registry, so they are rename-stable by construction and clearer than a fixture would make them. No wholesale migration is warranted.

The trust model that makes the above safe is unchanged: `ValidateSafetyInvariants` at load (the relocated anchor) **plus** the runtime artifact-backed layers (the `RoutingSignals` digest, the audit verdict routing, `CanTerminateEarly`, `ClampPlanToFloorWith`'s self-sealing evaluator, and the ship-time tree-SHA binding + EGPS `red_count==0`). The flow is data; the floor is a tested, phase-agnostic property.

### Implemented invariants vs. design intent (post-review reconciliation)

§4 lists the **design-intent** invariant set I1–I9; the validator as shipped implements a **subset**, and §"Why this is safe" overstated "structural attacks rejected at load by I1–I9." The accurate, audited status:

- **Implemented at load** (in `ValidateSafetyInvariants`): I8 branch-target legality; spine-edge legality (DDK-3); the floor-gate verdict constraint and `legal_successors` resolvability (DDK-4/DDK-5); I9 anchor reachability; the **floor-evaluator-existence** invariant (the operative half of "F⊆M" — a mandatory phase MUST gate on a shippable verdict, so dropping the evaluator from `mandatory_phases`, or giving it a presence-only gate, is rejected at load); and **evaluator graph-dominance** (the load-bearing half of I1/I2 — the floor evaluator SET must dominate the ship sink: with every shippable-verdict mandatory phase deleted from the graph, the last mandatory anchor is unreachable from the start node, so no start→ship path bypasses every evaluator). The evaluator and sink are identified phase-agnostically by role (the verdict gate / the last mandatory anchor), and the evaluator set is deleted as a whole (collective dominance — two evaluators each gating a disjoint path to ship still pass). The evaluator-existence and evaluator-dominance checks are judged only when the catalog is authoritative over the floor (describes ≥1 mandatory phase), so synthetic unit-test catalogs are not false-flagged.
- **NOT a load-time check; carried by the runtime backstop**: **all-anchor** path-dominance (I1 over a NON-evaluator anchor — e.g. a `legal_successors` `scout→build` that routes around a still-mandatory `triage` while the evaluator is still crossed). The shipped registry deliberately permits such edges; the linear spine is enforced at **runtime**, not by graph topology. So the load-time validator now proves the *evaluator* dominates ship, but intentionally does NOT require every anchor to dominate ship — doing so would reject the shipped registry's legitimate runtime-enforced spine. A config that strands a non-evaluator anchor's *artifact* is caught at **runtime** by `SpineSatisfiedUpTo` (ship requires every mandatory anchor's artifacts) and the ship phase's audit-binding (`ClassCycle` `verifyAuditBinding` — no auditor ledger entry ⇒ ship blocked). The floor therefore HOLDS for the evaluator at load AND for every anchor at runtime — defense-in-depth, with the load-bearing dominance now proven at load rather than deferred.
